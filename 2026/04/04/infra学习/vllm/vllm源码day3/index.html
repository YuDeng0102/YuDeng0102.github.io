Scheduler

Schedule是在迭代级别（iteration level）做出的。每一个调度步骤都对应模型的一次前向推断（forward pass）。因此，该方法会被引擎中的内部循环（busy loop）不断地反复调用。

本质上，调度器会生成一个 {req_id: num_tokens} 的字典，用来指定在当前的调度步骤中，需要为每个请求处理多少个 token。例如，对于全新的请求，num_tokens 的值可以达到该请求 prompt 的 token 数量；而对于正在通过自回归方式逐个生成新 token 的请求，该值通常为 1。除此之外，在涉及到分块预填充（chunked prefills）、前缀缓存（prefix caching）或投机采样（speculative decoding）等场景时，该值通常会介于上述两者（最大提示词长度与 1）之间。

此外，调度器还会返回有关各个单一请求或整个批次（batch）的有用数据。模型运行器（model runner）在为模型准备输入数据时，将会使用到这些信息。

核心成员

Scheduler 的核心函数就是schedule()，返回这一次执行的每个request以及对应的信息。

核心元素如下:

  • 两个队列：self.running和self.waiting, 前者是正在运行的请求，后者是等待调度的请求，running队列有两种类型，基于优先级的堆队列PriorityRequestQueue()和先进先出的FCFSRequestQueue()
  • self.connector = KVConnectorFactory.create_connector：负责不同worker之间kv-cache的通信
  • self.requests: dict[str, Request] = {} ，存储所有请求的字典 req_id -> Request
  • self.finished_req_ids: set[str] = set()：上一次和现在step()之间，完成的requests
  • self.kv_cache_manager = KVCacheManager：管理kv-cache的核心组件
  • 关于投机采用（speculative decoding）和encoder相关的信息后续再了解

schedule()

每轮执行一次 schedule()，返回 SchedulerOutput 给 Executor 用以触发一次 Worker 模型前向计算。可以被调度的requests有三种类别：

队列 来源状态 含义
scheduled_new_reqs WAITING 全新请求，第一次被调度
scheduled_running_reqs RUNNING 上轮就在跑，这轮继续
scheduled_resumed_reqs PREEMPTED 之前被踢出去过，这轮重新捡回来

之所以要区分waiting和preempted,是因为一个本来在 RUNNING 里正 Decode 吐字的请求，如果系统显存爆了，会被强行踢回 WAITING 队列（状态变为 PREEMPTED）。等显存宽裕了，它从 WAITING 里被重新捞出来时，由于之前的 KV Cache 已经被物理清空了，它不得不从头重新做一遍 Prefill（Recompute）。

schedule()的流程首先是一个while 循环，每次迭代选择一批本轮要执行的 request。

1. Running队列

目标：为正在运行的请求分配新的 token 所需的 KV cache block。

2. WAITING 队列（新来的/恢复的请求）

目标：寻找新请求填补剩余 token 预算。注意，只有在第一阶段没有发生抢占时，才尝试调度 waiting 请求。

3. 构建并返回 SchedulerOutput

目标：将前面两个阶段的请求打包供后续 Executor 使用。

_update_request_with_output()

负责在model_runner执行后，对单个request进行append生成的token以及检查这个request是否stop

Scheduler.update_from_output()

【调用时机】 发生在 model_runner.execute()（GPU 底层算力前向传播）刚刚执行完毕，且把结果（新生成的 token、logits 等数组）返回给主引擎之后。

【核心功能（剔除推测解码）】 这是一个“引擎后处理与状态结算”函数，主要完成 3 件事：

  1. 挂载记录新词（Bookkeeping）： 将 GPU 刚算出来的零散 new_token_ids 按 request_id 归属，依次添加到各个 Request 对象的自身历史序列中。
  2. 停机判定与显存释放（Stop & Free）： 对每个请求进行终局检查（是否产生 <EOS>，或触达 max_tokens 长度上限）。
    • 若否：保持 RUNNING 状态等待下一轮调度。
    • 若是：将状态置位 FINISHED，并立刻调用 kv_cache_manager.free 物理清空其占用的全部显存块，以腾出系统配额。
  3. 组装外部报文（Build Responses）： 把更新后的文本、logprobs 概率得分等包装成统一的 EngineCoreOutputs 结构，向上层（如 API 服务器流式端点）推流返回。

总结本质

有了 KV-Cache 之后，Attention 的计算从矩阵乘矩阵（Gemm, 复杂度 ），变成了向量乘矩阵（Gemv, 复杂度 ）。

虽然计算量大幅下降，但也带来了一个新的代价：随着生成序列 的变长， 和 这两个张量占用的显存空间会越来越大（这被称为 Memory Wall），这就是为什么像 vLLM 这样的工程框架需要极其复杂精明的机制（如 PagedAttention）来管理显存里的 KVCacheBlocks。

KVCacheManager

1. 核心定位 (Role & Purpose)

KVCacheManager 是 vLLM 调度系统（Scheduler）与底层物理显存（BlockPool / Coordinator）之间的**“高级显存大管家”**。 调度器不需要懂底层的 CUDA 内存是怎么划分的，它只管向 KVCacheManager 伸手要空间。管家负责进行前缀缓存查重、分配物理卡槽、回收垃圾和维护内存状态树。

2. 核心数据结构：KVCacheBlocks

为了将底层的显存块对象安全地打包返回给调度器，引入了 KVCacheBlocks （由 @dataclass 装饰的数据承载类）。

  • 设计亮点： 系统中存在大量“需要分配 0 个缓存块”的情况，为了避免引发严重的 Python GC（垃圾回收）卡顿，管家在初始化时利用 Python 嵌套元组（Tuple）的不可变性（Immutable）预先构造了一个全局单例的 empty_kv_cache_blocks，实现极致的安全复用。

3. 三大核心业务逻辑

① 查缓存与白嫖 (Prefix Caching)

核心函数：get_computed_blocks

  • 当一个新的请求进来时，管家拿着它的 Prompt 偷偷去公共显存池里“匹配查重”。
  • 发现可复用块（num_new_computed_tokens）： 发现跟历史某些 Prompt 吻合，直接白领对应的底层的 KV Cache Block（即 new_computed_block_list），不花算力就完成了进度。
  • 保留最后 1 个词机制： 为了强制模型在前向传播中吐出用于生成下一个词的 Logits 返回值，系统设置了 max_cache_hit_length = request.num_tokens - 1，强行保证最后 1 个 token 必须被计算。

② 开辟新显存卡槽 (Slot Allocation)

核心函数：allocate_slots 拼图式地组装一个请求该轮所需的全部显存空间，满足：

  • 过去已拥有的旧 Block 数量
  • 刚刚命中的白嫖前缀 Block 数量
  • 系统必须新开辟的纯空物理 Block (new_blocks)

③ 生命周期维护与优化 (Maintenance)

  • 内存释放： 通过 free 函数在请求结束时回收资源。
  • SWA（滑动窗口）极致抠门优化： 借助 remove_skipped_blocks 机制，在申请新内存前，先把由于超出 Attention 滑动窗口（比如前 4096 个词以外的词）导致连模型都不看的废弃老 Block 强行释放，反向挤出显存。
  • 算子级公共前缀加速： get_num_common_prefix_blocks 利用严格的引用计数（ref_cnt == 系统总活跃数），找出大家都在用的系统提示词（System Prompt）显存段，以便底层 CUDA (PagedAttention) 全局广播复用。

4. 关键变量防混淆指南

由于大量引入缓存机制，这里的 token 计算具有两面性：

  • num_new_tokens: 缓存里绝对没有，需要用真金白银的新显存，且必须走大模型跑一整套前向传播（Forward Pass）才能算出来的计算量负担。
  • num_computed_tokens / num_new_computed_tokens: 虽然名字里带 computed，但代表的是“前人已经算好躺在缓存里的 tokens”。
